# 1c-copilot MCP — Правила использования (для ИИ)

## Назначение

Ты подключён к MCP-серверу `1c-copilot`, который предоставляет доступ к специализированному ИИ-ассистенту **1С:Напарник** через API `code.1c.ai`. Через этот сервер ты можешь:
- Проверять код 1С на ошибки
- Проводить ревью кода по стилю и стандартам
- Генерировать и редактировать код 1С
- Искать информацию по платформе 1С, ИТС и типовым конфигурациям
- Получать консультации по разработке на 1С

## Ключевое ограничение

**НИКОГДА не передавай код 1С в параметре `code`, если есть возможность указать `file_path`.** Сервер сам прочитает код из файла — это экономит контекст и исключает ошибки копирования.

**Исключение:** только если код сгенерирован тобой или получен от пользователя в сообщении (не из файла).

## Версия платформы

**Сервер автоматически добавляет версию платформы 1С в каждый запрос** (переменная `ONEC_AI_PLATFORM_VERSION`, по умолчанию `8.3.25`). Не передавай версию в параметрах инструментов — это делает сервер. Если пользователь хочет сменить версию — подскажи указать её в настройках MCP-клиента.

## Подключение MCP-сервера

Чтобы пользователь мог использовать `1c-copilot` в своём MCP-клиенте, ему нужно добавить конфигурацию в настройки. Если пользователь спросит, как подключить — покажи этот пример.

**Требования:** сервер распространяется как один `.exe`-файл. Путь к нему нужно указать в настройках.

```json
{
  "mcpServers": {
    "1c-copilot": {
      "command": "полный\\путь\\к\\1c-copilot-mcp.exe",
      "env": {
        "ONEC_AI_TOKEN": "ваш_токен_с_code.1c.ai",
        "ONEC_AI_PLATFORM_VERSION": "8.3.25"
      }
    }
  }
}
```

**Где искать настройки MCP:**
- **Claude Desktop:** `%APPDATA%\Claude\claude_desktop_config.json`
- **VS Code (Cline, Continue):** настройки расширения → MCP Servers
- **Cursor:** `.cursor/mcp.json` в корне проекта
- **Windsurf:** `~/.codeium/windsurf/mcp_config.json`

**Единственный обязательный параметр — `ONEC_AI_TOKEN`.** Остальные опциональны и имеют разумные значения по умолчанию. Полный список переменных окружения:

| Переменная | По умолчанию | Описание |
|---|---|---|
| `ONEC_AI_TOKEN` | — | **Обязательный.** Токен API code.1c.ai |
| `ONEC_AI_BASE_URL` | `https://code.1c.ai` | Базовый URL API |
| `ONEC_AI_PLATFORM_VERSION` | `8.3.25` | Версия платформы для контекста |
| `ONEC_AI_UI_LANGUAGE` | `russian` | Язык интерфейса |
| `ONEC_AI_PROGRAMMING_LANGUAGE` | `1C (BSL)` | Язык программирования |
| `ONEC_AI_TIMEOUT` | `30` | Таймаут запроса (сек) |
| `ALLOWED_CODE_PATHS` | `.` | Разрешённые пути для чтения файлов (`;` для списка, `*` без ограничений) |
| `MAX_ACTIVE_SESSIONS` | `10` | Макс. активных сессий |
| `SESSION_TTL` | `3600` | Время жизни сессии (сек) |
| `ONEC_AI_LOG_FIELDS` | — | Поля для логирования: `prompt`, `reasoning`, `answer` через пробел или запятую. Если не задана — логирование отключено. Для логирования всего: `prompt reasoning answer` |

## Инструменты

### 1. `check_1c_code` — Комплексная проверка кода

**Что делает:** проверяет код 1С по четырём аспектам: синтаксис, логика, производительность, архитектура. Возвращает проблемы с приоритетами (критические/серьёзные/рекомендательные) и конкретные исправления.

**Когда использовать:**
- Пользователь просит проверить код на ошибки
- Пользователь просит найти баги или проблемы в коде
- Пользователь спрашивает "правильно ли написан этот код?"
- Ты написал код 1С и хочешь его проверить перед тем как показать пользователю

**Когда НЕ использовать:**
- Для оценки стиля и читаемости (используй `review_1c_code`)
- Для поиска документации (используй `search_1c_knowledge`)
- Для объяснения синтаксиса (используй `explain_1c_syntax`)

**Параметры:**

| Параметр | Обязательный | Правило заполнения |
|---|---|---|
| `code` | если нет file_path | Передавай inline-код ТОЛЬКО если код не из файла |
| `file_path` | если нет code | **Предпочитаемый способ.** Путь к файлу относительно рабочей директории проекта |
| `start_line` | нет | Номер строки начала (1-based). Указывай, если нужно проверить фрагмент |
| `end_line` | нет | Номер строки конца (1-based). Указывай вместе с start_line |

**Примеры:**

```json
// ПРОВЕРКА ЦЕЛОГО ФАЙЛА — предпочитаемый способ
{
  "file_path": "src/Documents/ЗаказКлиента/Forms/ФормаДокумента/Module.bsl"
}

// ПРОВЕРКА КОНКРЕТНЫХ СТРОК
{
  "file_path": "src/CommonModules/ОбменДанными/Module.bsl",
  "start_line": 45,
  "end_line": 120
}

// ПРОВЕРКА INLINE-КОДА (только если код не в файле)
{
  "code": "Процедура Проверить(Отказ)\n  Движения.Товары.Записать()\nКонецПроцедуры"
}
```

### 2. `review_1c_code` — Ревью по стилю и стандартам

**Что делает:** оценивает качество оформления и методологию. НЕ ищет ошибки в логике/синтаксисе. Проверяет: стиль (именование PascalCase, форматирование, комментарии), стандарты (методическое пособие, структура модулей), лучшие практики (DRY, читаемость, модульность). Возвращает итоговый вердикт: хорошо/средне/плохо.

**Когда использовать:**
- Пользователь просит code review
- Пользователь спрашивает "хорошо ли написан код?", "соответствует ли стандартам?"
- Пользователь хочет улучшить читаемость и поддерживаемость
- Перед рефакторингом — оценить текущее состояние кода

**Когда НЕ использовать:**
- Для поиска ошибок и багов (используй `check_1c_code`)
- Для изменения кода (используй `modify_1c_code`)

**Параметры:**

| Параметр | Обязательный | Правило заполнения |
|---|---|---|
| `code` | если нет file_path | Inline-код (только если не из файла) |
| `file_path` | если нет code | **Предпочитаемый способ.** Путь к файлу |
| `start_line` | нет | Начальная строка фрагмента |
| `end_line` | нет | Конечная строка фрагмента |
| `context` | нет | Имя конфигурации или стандартов, которые нужно учесть |

**Примеры:**

```json
{
  "file_path": "src/Catalogs/Номенклатура/Forms/ФормаЭлемента/Module.bsl",
  "context": "Управление торговлей, редакция 11"
}
```

### 3. `modify_1c_code` — Редактирование кода

**Что делает:** изменяет существующий код 1С согласно инструкции. Возвращает ПОЛНЫЙ исправленный код (не только изменённый фрагмент). Ключевое правило: менять только то, что запрошено.

**Когда использовать:**
- Рефакторинг существующего кода
- Исправление найденных ошибок
- Добавление функционала в существующий код
- Применение рекомендаций из `check_1c_code` или `review_1c_code`

**Когда НЕ использовать:**
- Для написания кода с нуля (используй `generate_1c_code`)

**Параметры:**

| Параметр | Обязательный | Правило заполнения |
|---|---|---|
| `code` | если нет file_path | Исходный inline-код |
| `file_path` | если нет code | **Предпочитаемый способ.** Путь к файлу |
| `start_line` | нет | Начальная строка для частичного редактирования |
| `end_line` | нет | Конечная строка |
| `instruction` | да | **Что именно изменить.** Пиши чётко |

**Примеры:**

```json
{
  "file_path": "src/Documents/РеализацияТоваров/Forms/ФормаДокумента/Module.bsl",
  "start_line": 80,
  "end_line": 150,
  "instruction": "Заменить цикл обхода ВыборкаДетальныеЗаписи.Следующий() на пакетный запрос с временной таблицей. Сохранить логику расчёта сумм."
}
```

### 4. `generate_1c_code` — Генерация кода с нуля

**Что делает:** пишет код 1С по описанию задачи. Возвращает готовый код в ````bsl` блоке.

**Когда использовать:**
- Создание новой процедуры/функции по ТЗ
- Написание модуля обработчика с нуля
- Генерация запроса по описанию

**Параметры:**

| Параметр | Обязательный | Правило заполнения |
|---|---|---|
| `task` | да | Описание задачи: что должно быть реализовано |
| `context` | нет | Контекст: в какой конфигурации работаем, какие объекты метаданных доступны, версия платформы |
| `output_format` | нет | `full` (по умолчанию), `module`, `procedure`, `function` |

**Примеры:**

```json
{
  "task": "Функция расчёта скидки: если сумма заказа больше 10000 — скидка 10%, если больше 50000 — 15%. Параметр: СуммаЗаказа (Число). Возвращает: Число (процент скидки).",
  "context": "Конфигурация Управление торговлей 11, платформа 8.3.25",
  "output_format": "function"
}
```

### 5. `search_1c_knowledge` — Поиск знаний 1С

**Что делает:** ищет информацию по ИТС, платформе, типовым конфигурациям и **Библиотеке стандартных подсистем (БСП)** через знания модели. Возвращает структурированный ответ с указанием источников (ИТС / платформа / конфигурация / БСП). **Это НЕ веб-поиск** — модель отвечает на основе своего обучения.

**Когда использовать:**
- Вопросы об устройстве типовой конфигурации
- Поиск методических рекомендаций
- Вопросы о механизмах платформы
- **Описание функций БСП, их параметров и назначения**
- **Исходные тексты общих модулей БСП, обработчиков, алгоритмов**
- "Как в 1С реализован X?", "Где в типовой находится Y?"

**Параметры:**

| Параметр | Обязательный | Правило заполнения |
|---|---|---|
| `query` | да | Поисковый запрос на русском языке |
| `configuration` | нет | Имя конфигурации для фокусировки |
| `platform_version` | нет | Версия платформы для фокусировки |

**Примеры:**

```json
{
  "query": "Как работает механизм отражения зарплаты в регламентированном учёте?",
  "configuration": "Зарплата и управление персоналом 3.1",
  "platform_version": "8.3.25"
}
```

### 6. `ask_1c_ai` — Консультация по 1С

**Что делает:** универсальный инструмент — задать вопрос 1С:Напарник. Режимы: общий ответ, развёрнутая консультация, глубокий экспертный разбор.

**Когда использовать:**
- Общие вопросы о 1С (не подходящие под другие инструменты)
- Нужна консультация с обоснованием и альтернативами
- Архитектурные решения: "как лучше спроектировать X?"
- Миграция, обновление, выбор между подходами

**Параметры:**

| Параметр | Обязательный | Правило заполнения |
|---|---|---|
| `question` | да | Вопрос |
| `consultation_type` | нет | `general`, `consulting`, `expertise` |
| `create_new_session` | нет | `true` если нужен новый разговор без контекста |

**Примеры:**

```json
{
  "question": "Как правильнее организовать обмен данными между УТ 11 и БП 3.0 для нетиповых справочников?",
  "consultation_type": "consulting"
}
```

### 7. `explain_1c_syntax` — Объяснение синтаксиса

**Что делает:** объясняет синтаксис, конструкцию языка или объект метаданных 1С: назначение, примеры, особенности, типичные ошибки.

**Когда использовать:**
- Пользователь спрашивает "как работает X?"
- Нужно объяснить конструкцию языка
- Пользователь не понимает синтаксис конкретного оператора

**Параметры:**

| Параметр | Обязательный | Правило заполнения |
|---|---|---|
| `syntax_element` | да | Что объяснить |
| `context` | нет | Дополнительный контекст |

**Пример:**

```json
{
  "syntax_element": "Для Каждого ... Цикл ... КонецЦикла",
  "context": "обход таблицы значений"
}
```

### 8. `explore_1c_codebase` — Исследование кодовой базы

**Что делает:** ищет объекты метаданных, их связи и зависимости в конфигурации 1С.

**Отличие от `search_1c_knowledge`:** `explore_1c_codebase` — для вопросов о том, КАК устроена конфигурация (какие объекты, где используются, связи между ними). `search_1c_knowledge` — для вопросов о том, КАК РАБОТАЕТ механизм (документация, методика).

**Когда использовать:**
- "Найди где используется справочник X"
- "Какие объекты связаны с документом Y?"
- "Покажи структуру модуля Z"

**Параметры:**

| Параметр | Обязательный | Правило заполнения |
|---|---|---|
| `query` | да | Что найти или исследовать |
| `code` | нет | Фрагмент кода для анализа его связей (только если код не в файле) |

**Пример:**

```json
{
  "query": "Какие реквизиты у документа РеализацияТоваровУслуг в БП 3.0?"
}
```

**Когда использовать:**
- Создана новая функция — нужно задокументировать
- Пользователь просит "добавь комментарий к этой процедуре"

**Параметры:**

| Параметр | Обязательный | Правило заполнения |
|---|---|---|
| `code` | если нет file_path | Inline-код процедуры/функции |
| `file_path` | если нет code | **Предпочитаемый способ.** Путь к файлу |
| `start_line` | нет | Строка с процедурой/функцией |
| `end_line` | нет | Конец процедуры/функции |

## Правила выбора: file_path vs inline code

| Ситуация | Что передавать |
|---|---|
| Код есть в файле проекта | `file_path` — всегда |
| У тебя есть путь из предыдущих действий | `file_path` |
| Пользователь указал путь | `file_path` |
| Код только что сгенерирован тобой | `code` |
| Код вставлен пользователем в сообщение | `code` |
| Гипотетический фрагмент "что если..." | `code` |

### Указание диапазона строк
Используй `start_line`/`end_line` когда:
- Нужно проверить/отредактировать конкретный метод, а не весь модуль
- Пользователь указал конкретные строки
- Код проблемы находится в определённой области файла

**Важно: всегда захватывай документирующий комментарий перед функцией/процедурой.** Если передаёшь диапазон строк для анализа функции или процедуры, начинай `start_line` со строки, предшествующей её документирующему комментарию (блок `//` перед объявлением). Это даёт модели контекст: описание параметров, возвращаемого значения и назначения кода — что делает ревью и проверку качественнее.

```
// Пример: функция начинается на строке 15, комментарий — на строках 10-14
// значит start_line = 10, end_line = 54 (включая комментарий)
// 10: // Возвращает токен авторизации API СДЭК
// 11: // Параметры:
// 12: //  Соединение - HTTPСоединение - ...
// 13: // ...
// 14: //
// 15: Функция ПолучитьТокенСДЭК(Соединение)
// ...
// 54: КонецФункции
```

НЕ используй диапазон когда:
- Нужно проанализировать весь модуль/файл
- Зависимости кода выходят за пределы фрагмента

## Типичные сценарии (workflows)

### Сценарий 1: Пользователь просит проверить код

```
1. check_1c_code(file_path="путь/к/файлу.bsl")
2. Проанализируй ответ: что критично, что рекомендательно
3. Если есть ошибки → modify_1c_code с исправлениями
4. Покажи пользователю: список проблем + исправленный код
```

### Сценарий 2: Пользователь просит написать новую функцию

```
1. search_1c_knowledge(query="как в типовой реализован X", configuration="ИмяКонфигурации")
2. generate_1c_code(task="описание задачи", context="контекст из п.1")
3. check_1c_code(code="сгенерированный код") — проверить перед показом
4. review_1c_code(code="сгенерированный код") — оценить стиль
5. generate_docstring(code="сгенерированный код") — добавить комментарий
6. Показать пользователю финальный код с комментарием
```

### Сценарий 3: Пользователь просит отрефакторить код

```
1. review_1c_code(file_path="путь/к/файлу.bsl") — понять текущее состояние
2. check_1c_code(file_path="путь/к/файлу.bsl") — найти ошибки
3. modify_1c_code(file_path="путь/к/файлу.bsl", instruction="чёткая инструкция на основе п.1 и п.2")
4. check_1c_code(code="исправленный код") — проверить результат
5. Показать пользователю результат
```

### Сценарий 4: Пользователь просит написать код (сразу с проверкой)

```
1. generate_1c_code(task="описание") — написать код
2. check_1c_code(code="сгенерированный код") — проверить на ошибки
3. Если найдены ошибки → modify_1c_code для их исправления
4. Показать пользователю финальный код
```

## Важные замечания

1. **Сервер не хранит код.** Каждый вызов инструмента — независимый запрос к API 1С:Напарник. Если ты хочешь передать результат одного вызова в другой — сохраняй его в переменную и передавай в `code`.

2. **Не дублируй запросы.** Если `check_1c_code` уже вернул список ошибок, не вызывай его снова для того же кода — сразу переходи к `modify_1c_code`.

3. **Не злоупотребляй `ask_1c_ai`.** Для специализированных задач (проверка, ревью, поиск) используй соответствующие инструменты — они дают более качественные и структурированные ответы.

4. **`review_1c_code` не проверяет ошибки.** Если пользователь просит "проверь код на ошибки и заодно оцени стиль" — вызови ОБА инструмента: сначала `check_1c_code`, затем `review_1c_code`.

5. **Сессии.** Сервер автоматически переиспользует сессии 1С:Напарник для поддержания контекста разговора. Чтобы начать разговор заново — передай `create_new_session: true`. Этот параметр работает ТОЛЬКО с инструментом `ask_1c_ai`.

6. **Диапазоны строк.** `start_line` и `end_line` — 1-based (первая строка = 1). Если `start_line=0` — читается с начала файла. Если `end_line=0` — читается до конца.

7. **PascalCase.** В 1С используется PascalCase (КаждоеСловоСЗаглавной) для всех идентификаторов: переменных, процедур, функций, параметров. Не используй camelCase.

8. **Формулируй запросы конкретно.** Для `check_1c_code` и `review_1c_code` указывай, на что обратить особое внимание. Для `search_1c_knowledge` — какой именно аспект интересует.
