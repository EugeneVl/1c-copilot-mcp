# 1c-copilot MCP — Правила использования (для ИИ)

## Назначение

Ты подключён к MCP-серверу `1c-copilot`, который предоставляет доступ к специализированному ИИ-ассистенту **1С:Напарник** через API `code.1c.ai`. Через этот сервер ты можешь:
- Проверять код 1С (синтаксис, логика, производительность, архитектура, стиль, стандарты, лучшие практики)
- Искать информацию по платформе 1С, ИТС и типовым конфигурациям
- Получать консультации по разработке на 1С

## Ключевое ограничение

**НИКОГДА не передавай код 1С в параметре `code`, если есть возможность указать `file_path`.** Сервер сам прочитает код из файла — это экономит контекст и исключает ошибки копирования.

**Исключение:** только если код получен от пользователя в сообщении (не из файла).

## Правила выбора инструмента (алгоритм для ИИ)

Когда пользователь просит что-то сделать с кодом 1С, выбирай инструмент по этой логике:

```
Пользователь просит:
│
├─ "проверь код на ошибки" / "найди баг" / "правильно ли?" / "сделай code review" / "оцени стиль"
│  └─ check_1c_code
│
├─ "как работает X в 1С?" / "объясни синтаксис"
│  └─ explain_1c_syntax
│
├─ "найди в ИТС" / "как устроено в типовой" / "документация по платформе"
│  └─ search_1c_knowledge
│
├─ "найди где используется" / "какие связи у объекта"
│  └─ explore_1c_codebase
│
└─ любой другой вопрос о 1С / консультация
   └─ ask_1c_ai

─────────────
Перед вызовом любого инструмента определи способ передачи кода:

Код есть в файле? ──► file_path (всегда)
      │
      └─ Нет: код получен от пользователя
         или гипотетический фрагмент? ──► code
```

## Инструменты

### 1. `check_1c_code` — Комплексная проверка кода

**Что делает:** проверяет код 1С по семи аспектам: синтаксис, логика, производительность, архитектура, стиль оформления, стандарты разработки и лучшие практики. Включает экспертный разбор: архитектурные последствия, влияние на производительность, риски при обновлении конфигурации, соответствие методическим рекомендациям 1С. Возвращает проблемы с приоритетами (критические/серьёзные/рекомендательные) и номерами строк, где найдена проблема. Краткие исправления (1-2 строки) могут быть указаны в скобках, полный исправленный код не выводится. Итоговый вердикт: оценка качества кода (хорошо/средне/плохо).

**Когда использовать:**
- Пользователь просит проверить код на ошибки
- Пользователь просит найти баги или проблемы в коде
- Пользователь спрашивает "правильно ли написан этот код?"
- Пользователь просит code review / оценить стиль / проверить соответствие стандартам

**Когда НЕ использовать:**
- Для поиска документации (используй `search_1c_knowledge`)
- Для объяснения синтаксиса (используй `explain_1c_syntax`)

**Параметры:**

| Параметр | Обязательный | Правило заполнения |
|---|---|---|
| `code` | если нет file_path | Передавай inline-код ТОЛЬКО если код не из файла |
| `file_path` | если нет code | **Предпочитаемый способ.** Путь к файлу относительно рабочей директории проекта |
| `start_line` | нет | Номер строки начала (1-based). Указывай, если нужно проверить фрагмент |
| `end_line` | нет | Номер строки конца (1-based). Указывай вместе с start_line |
| `context` | нет | Имя конфигурации или стандартов, которые нужно учесть |

---

### 2. `search_1c_knowledge` — Поиск знаний 1С

**Что делает:** **ВАЖНО: это НЕ веб-поиск.** Модель отвечает на основе своего обучения, не обращаясь к интернету. Ищет информацию по ИТС, платформе, типовым конфигурациям и Библиотеке стандартных подсистем (БСП) через знания модели.

**Когда использовать:**
- Вопросы об устройстве типовой конфигурации
- Поиск методических рекомендаций
- Вопросы о механизмах платформы
- Описание функций БСП, их параметров и назначения
- Исходные тексты общих модулей БСП, обработчиков, алгоритмов
- "Как в 1С реализован X?", "Где в типовой находится Y?"

**Параметры:**

| Параметр | Обязательный | Правило заполнения |
|---|---|---|
| `query` | да | Поисковый запрос на русском языке |
| `configuration` | нет | Имя конфигурации для фокусировки |
| `platform_version` | нет | Версия платформы для фокусировки |

---

### 3. `ask_1c_ai` — Консультация по 1С

**Что делает:** универсальный инструмент — задать вопрос 1С:Напарник. Режимы: общий ответ (2-3 предложения), развёрнутая консультация, глубокий экспертный разбор.

**Когда использовать:**
- Общие вопросы о 1С (не подходящие под другие инструменты)
- Нужна консультация с обоснованием и альтернативами
- Архитектурные решения: "как лучше спроектировать X?"
- Миграция, обновление, выбор между подходами

**Параметры:**

| Параметр | Обязательный | Правило заполнения |
|---|---|---|
| `question` | да | Вопрос |
| `consultation_type` | нет | `general`, `consulting`, `expertise` |

---

### 4. `explain_1c_syntax` — Объяснение синтаксиса

**Что делает:** объясняет синтаксис, конструкцию языка или объект метаданных 1С. Для простых вопросов — кратко (2-3 предложения), для сложных — с примерами кода.

**Когда использовать:**
- Пользователь спрашивает "как работает X?"
- Нужно объяснить конструкцию языка
- Пользователь не понимает синтаксис конкретного оператора

**Параметры:**

| Параметр | Обязательный | Правило заполнения |
|---|---|---|
| `syntax_element` | да | Что объяснить |
| `context` | нет | Дополнительный контекст |

---

### 5. `explore_1c_codebase` — Исследование кодовой базы

**Что делает:** ищет объекты метаданных, их связи и зависимости в конфигурации 1С. Ответ — маркированный список, каждый пункт: имя объекта и краткое описание (1 предложение).

**Отличие от `search_1c_knowledge`:** `explore_1c_codebase` — для вопросов о том, КАК устроена конфигурация (какие объекты, где используются, связи между ними). `search_1c_knowledge` — для вопросов о том, КАК РАБОТАЕТ механизм (документация, методика).

**Когда использовать:**
- "Найди где используется справочник X"
- "Какие объекты связаны с документом Y?"
- "Покажи структуру модуля Z"

**Параметры:**

| Параметр | Обязательный | Правило заполнения |
|---|---|---|
| `query` | да | Что найти или исследовать |
| `code` | нет | Фрагмент кода для анализа его связей (только если код не в файле) |

## Правила выбора: file_path vs inline code

| Ситуация | Что передавать |
|---|---|
| Код есть в файле проекта | `file_path` — всегда |
| У тебя есть путь из предыдущих действий | `file_path` |
| Пользователь указал путь | `file_path` |
| Код вставлен пользователем в сообщение | `code` |
| Гипотетический фрагмент "что если..." | `code` |

### Указание диапазона строк
Используй `start_line`/`end_line` когда:
- Нужно проверить конкретный метод, а не весь модуль
- Пользователь указал конкретные строки
- Код проблемы находится в определённой области файла

**Важно: всегда захватывай документирующий комментарий перед функцией/процедурой.** Если передаёшь диапазон строк для анализа функции или процедуры, начинай `start_line` со строки, предшествующей её документирующему комментарию (блок `//` перед объявлением). Это даёт модели контекст: описание параметров, возвращаемого значения и назначения кода — что делает ревью и проверку качественнее.

```
// Пример: функция начинается на строке 15, комментарий — на строках 10-14
// значит start_line = 10, end_line = 54 (включая комментарий)
// 10: // Возвращает токен авторизации API СДЭК
// 11: // Параметры:
// 12: //  Соединение - HTTPСоединение - ...
// 13: // ...
// 14: //
// 15: Функция ПолучитьТокенСДЭК(Соединение)
// ...
// 54: КонецФункции
```

НЕ используй диапазон когда:
- Нужно проанализировать весь модуль/файл (например: «проверь весь модуль формы документа»)
- Зависимости кода выходят за пределы фрагмента (например: функция внутри диапазона вызывает `ОбщийМодуль.СерверныйОбщийМодуль.Рассчитать(...)`, а тот модуль не передан на проверку)
- Есть несколько разнесённых по файлу процедур/функций, которые нужно проверить вместе — лучше передать весь файл

## Типичные сценарии (workflows)

### Сценарий 1: Пользователь просит проверить код

```
1. check_1c_code(file_path="путь/к/файлу.bsl")
2. Проанализируй ответ: что критично, что рекомендательно
3. Покажи пользователю список проблем с номерами строк
```

### Сценарий 2: Пользователь просит консультацию по 1С

```
1. ask_1c_ai(question="вопрос", consultation_type="expertise")
2. Покажи пользователю результат
```

### Сценарий 3: Пользователь просит найти информацию

```
1. search_1c_knowledge(query="запрос", configuration="ИмяКонфигурации")
2. Покажи пользователю результат
```

## Важные замечания

1. **Сервер не хранит код.** Каждый вызов инструмента — независимый запрос к API 1С:Напарник.

2. **Не злоупотребляй `ask_1c_ai`.** Для специализированных задач используй соответствующие инструменты — они дают более качественные и структурированные ответы.

3. **`check_1c_code` делает полную проверку.** Проверяет синтаксис, логику, производительность, архитектуру, стиль, стандарты и лучшие практики в одном вызове. Не нужно вызывать отдельный инструмент для code review.

4. **Сессии.** Сервер автоматически переиспользует сессии 1С:Напарник для поддержания контекста разговора.

5. **Диапазоны строк.** `start_line` и `end_line` — 1-based (первая строка = 1). Если `start_line=0` — читается с начала файла. Если `end_line=0` — читается до конца.

6. **PascalCase.** В 1С используется PascalCase (КаждоеСловоСЗаглавной) для всех идентификаторов: переменных, процедур, функций, параметров. Не используй camelCase.

7. **Формулируй запросы конкретно.** Для `check_1c_code` указывай, на что обратить особое внимание. Для `search_1c_knowledge` — какой именно аспект интересует.

8. **Параллельные вызовы.** Все инструменты 1c-copilot можно вызывать параллельно, в том числе один и тот же инструмент одновременно с разными параметрами.

9. **Обработка ошибок.** Если вызов любого инструмента 1c-copilot вернул ошибку (нет ответа, таймаут, ошибка соединения) — MCP-сервер недоступен. Используй другие доступные способы решения задачи.
